How Tor Metrics preserves user privacy

The metrics on this website are based on different data sources in the Tor network. Some of these data sources are not sensitive at all, like properties and capabilities of a relay. Others are more sensitive, like statistics on fetched directory listings by country. Yet others are simply too sensitive to gather at all, like contents of unencrypted connections leaving the Tor network, so we don't have metrics on those.

We wrote a research paper where we describe how we measure potentially sensitive data in the Tor network. Whenever we plan to add new data, this plan needs to go through a rigorous process of writing a proposal document, which is usually discussed on the public development list, and publicly reviewing code patches on the bug tracker. Furthermore, as a core principle, we only use data for metrics that have been made publicly available: if the raw data are too sensitive to publish, then we shouldn't even publish aggregate statistics of it.

Contributing to Tor Metrics

We welcome all kinds of contributions that help people understand the Tor network better and at the same time do not put Tor users at risk. Here's an overview of possible contributions to Tor Metrics:


  1. Create a new metric: Did you come up with an idea to visualize existing Tor network data in a way that is not available on Tor Metrics yet? We can easily host non-interactive graphs or tables on this website, and we can add links to possibly interactive graphs or other metrics hosted by yourself. You're free to use any pre-aggregated data provided by Tor Metrics, Tor network status data provided by the Onionoo service, or raw descriptors provided by the CollecTor service.
  2. Run custom experiment: Are you considering to run an experiment on the Tor network and to share your data and findings? We can host both your data and your visualizations on this website and make them available to a wider audience. But let us emphasize the "do not put Tor users at risk" part again here. If you're unsure whether your experiment might be used to harm users, please talk to us before starting it.
  3. Add/update metric generated by Tor Metrics: We happily accept contributions to the Metrics code base that add or update metrics provided on this website. This requires effort on your side for writing a patch and on our side to review it, but the result is a fully integrated, interactive metric and an honorable mention for creating it. We're happy to provide assistance in writing such a patch.
  4. Add/update data-aggregating script run by Tor Metrics: If you're really brave, you may want to contribute a patch to the data-processing scripts that update the data provided on this website. The cool thing is that future metrics may be based on your data, so this is really the most effective way of contributing to Tor Metrics. And there's an honorable mention in it, too. Please come talk to us if this sounds like a fun adventure to you.

The general approach for contributing to Tor Metrics is to open a ticket in the issue tracker using the "Metrics Website" component and discuss your contribution there. And no worries, if you have a good idea but can't implement it on your own, please suggest it anyway! But please understand that adding a new graph or table may be harder than it seems and take longer than you may expect.

How to cite Tor Metrics in a research paper

If you're writing a research paper that uses data or metrics from this website, attribution is appreciated. Most people cite the following research paper:

Karsten Loesing, Steven J. Murdoch, and Roger Dingledine. A Case Study on Measuring Statistical Data in the Tor Anonymity Network. In Proceedings of the Workshop on Ethics in Computer Security Research (WECSR 2010), Tenerife, Canary Islands, Spain, January 2010.

@inproceedings{wecsr10measuring-tor,
  title = {A Case Study on Measuring Statistical Data in the {T}or Anonymity Network},
  author = {Karsten Loesing and Steven J. Murdoch and Roger Dingledine},
  booktitle = {Proceedings of the Workshop on Ethics in Computer Security Research (WECSR 2010)},
  year = {2010},
  month = {January},
  location = {Tenerife, Canary Islands, Spain},
  publisher = {Springer},
  series = {LNCS},
}

Alternatively, you can of course cite this website directly. Be careful with links to specific metrics though. In most cases, citing https://metrics.torproject.org/ should be sufficient.
